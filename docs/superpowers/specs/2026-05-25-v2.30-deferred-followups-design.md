## 四、本轮各项的落地设计

### 4.1 HX-9 — 优雅退出 flush

**问题**：`server/db.js#logsDB.add` 每 10 条 save / `livenessLogsDB.add` 每 5 条
save。崩溃 / Ctrl+C 时 0-9 条日志丢失。`save()` 现在是异步队列（v2.29），即便
mutation 同步入内存，队尾还可能没刷盘。

**实现**：在 `server/index.js` 顶部注册：

```js
let _shuttingDown = false;
async function gracefulShutdown(signal) {
  if (_shuttingDown) return;
  _shuttingDown = true;
  console.log(`[shutdown] received ${signal}, flushing…`);
  try {
    const db = require('./db');
    if (db.logsDB?.flush) db.logsDB.flush();      // marks all pending log entries dirty
    if (db.save?.flush) await db.save.flush();    // wait for atomic write to land
  } catch (e) { console.error('[shutdown] flush failed:', e.message); }
  try {
    if (typeof engine !== 'undefined' && engine?.stop) await engine.stop();
  } catch {}
  process.exit(0);
}
process.on('SIGINT', () => gracefulShutdown('SIGINT'));
process.on('SIGTERM', () => gracefulShutdown('SIGTERM'));
```

`save.flush()` 已在 v2.29 提供。logsDB.flush() 已存在（强制 save）。改动只在
`server/index.js`，约 25 行。

**测试**：新增 `server/__tests__/shutdown-flush.test.js`，spawn 一个 node 子进
程跑一个 `setTimeout + writeLog + signal`，验证 SIGINT 后子进程退出码为 0 且
on-disk db 含日志条目。

### 4.2 HX-13 — Windows Python 子进程 tree-kill

**问题**：`protocol-engine.js` 的 `py.kill()` Windows 上只 SIGTERM
`python.exe`，curl_cffi 内部 spawn 出来的 C 子进程不接受信号，会变僵尸。

**实现**：写一个 `server/process-utils.js` 暴露 `killTree(pid)`：

```js
const { spawnSync } = require('child_process');
exports.killTree = function (pid) {
  if (!pid) return;
  if (process.platform === 'win32') {
    spawnSync('taskkill', ['/PID', String(pid), '/T', '/F'], { stdio: 'ignore' });
  } else {
    try { process.kill(-pid, 'SIGKILL'); } catch {}  // requires detached + process group
    try { process.kill(pid, 'SIGKILL'); } catch {}
  }
};
```

`protocol-engine.js#stop()` 改用 `killTree(py.pid)` 代替 `py.kill()`。`server/
engine.js` 的 chromeProc 同样可走 `killTree`（防 Chrome renderer / GPU 孙进
程残留）。

**测试**：新增 `server/__tests__/process-utils.test.js`：spawn 一个 node 子进
程让它再 spawn 一个 grandchild（同样 node），用 `killTree(parentPid)` 杀掉
parent，验证 grandchild 也终止（用 `pidExists()` 轮询，含 600ms 超时）。仅
Windows 启用该测试用例；非 Windows 跳过。

### 4.3 HX-16 — `/api/health`

**问题**：运维 / 远程探活无端点；目前要么读 server.log 要么访问 `/` 看 502。

**实现**：新加 `server/routes/health.js`：

```js
router.get('/', (req, res) => {
  let dbOk = false;
  try { require('../db').accountsDB.list(); dbOk = true; } catch {}
  let proxyState = null;
  try {
    const s = require('../proxy').getState();
    proxyState = {
      enabled: s.enabled,
      jpEnabled: s.jp.enabled,
      currentNode: s.currentNode,
      jpCurrentNode: s.jp.currentNode,
      available: s.available,
      jpAvailable: s.jp.available,
    };
  } catch {}
  const engineStatus = (typeof engine !== 'undefined' && engine?.getStatus) ? engine.getStatus() : 'idle';
  res.json({
    ok: dbOk,
    db: dbOk ? 'ok' : 'error',
    proxy: proxyState,
    engine: engineStatus,
    uptimeSec: Math.floor(process.uptime()),
    version: require('../../package.json').version,
  });
});
```

`server/index.js` mount 到 `/api/health`。`engine` 在 routes/execute.js 是
module-level 单例；需要把它暴露出来以便 health 路由读 — 比 require cycle 更
干净的办法：把 engine 实例挪到 `server/engine-singleton.js`，routes 之间通过
`getEngine()` 函数获取。范围控制不动业务管道的前提下，仅做读封装。

**测试**：新增 `server/__tests__/health-endpoint.test.js`，用 supertest 不可行
（没装），改写为模块级直接 require routes/health.js 并 mock req/res。

### 4.4 PX-7 — 主动健康检查

**问题**：被动信号导致首批 3 个账号必失才能识别死节点；TTL 到期回到候选池
又浪费 3 个。

**实现**：`server/proxy/health-probe.js` 新模块，导出 `probeAllNodes(channel,
{ port })`，通过 Clash API 把 selector 临时切到每个节点，调用一次轻量 HTTP
（HEAD `https://api.ipify.org` via local proxy port），15s timeout，结果记到
`_state.failCount`/`failReasons`（不直接拉黑，只标"低优"）。

**触发**：`server/proxy/index.js#refresh()` 末尾，**异步**触发 `probeAllNodes`
（不 await，不阻塞 refresh 返回），完成后通过 `_state.healthChecked = true` +
推一条 `[Proxy] health probe: 12/15 alive` 日志。`rotate()` 在 prefer 时跳过
`failCount > 0` 的（已有逻辑），所以自动获益。

**配置**：`config.proxy.activeHealthCheck` 默认 `true`，关掉退化为 v2.29 行
为。配置入 `Config.vue` 代理 tab 一个 switch。

**测试**：新增 `server/proxy/__tests__/health-probe.test.js`，mock Clash API
+ http.request，验证 probe 顺序、超时、跳过 manual 拉黑节点。

### 4.5 FX-15 — Accounts 长字段折叠

**问题**：主表 4 个 monospace 字段（密码 / TOTP / Client ID / Refresh Token）
单条约占 600px，1366px 屏幕严重溢出。

**实现**：

- 移除"TOTP 密钥"、"Client ID"、"Refresh Token" 3 列。
- 保留"密码"列（最常需要的）。
- 加 `type="expand"` 列：点击展开时显示完整凭证表格 + 每字段的复制按钮。
- 主表新增"凭证"列（width 80），用三个小 icon + tooltip 显示是否设置（绿点 /
  灰点），点击 icon 展开行。

**好处**：主表瘦身约 350px；凭证细节仍可一键查阅；移动端不再溢出。

### 4.7 FX-13 — 通知中心

**问题**：ElMessage 一次性 toast 3s 闪过，错误易错过；运营没有"我刚才看到啥错"
的回看能力。

**实现**：

- 新加 `web/src/notifications.js`：`reactive({ items: [], unread: 0 })`；
  `pushNotification({ level, title, message })`；最多保留 100 条。
- AppLayout 顶栏铃铛 icon + unread badge；点击打开 `el-drawer` 列出最近 50
  条，按 level 染色，支持"全部已读 / 清空"。
- 全局拦截 ElMessage / ElMessageBox 的 type=error 调用：写一个 thin wrapper
  `notifyError(message, opts)`，在 `web/src/api.js` 的 axios interceptor 里
  把所有非 2xx 响应自动入通知中心。
- 现有 `ElMessage.error(...)` 调用**保留不动**（兼容）；新增的 axios 错误路径
  额外推到通知中心，不双弹（toast + 通知中心 = 一个交互，不是两个）。

### 4.8 FX-5 — 批量删除：后端 endpoint + 前端进度

**问题**：`Accounts.vue#delSelected` 串行 N 次 HTTP，N=200 时卡 30s 不知进度。

**实现**：

- 后端新增 `POST /api/accounts/batch-delete`，body `{ emails: string[] }`。
  accountsDB 加 `bulkDelete(emails)`：循环 `db.run("DELETE …")`，单次
  `save()`（sql.js 没有真事务，但单次 save 已经原子落盘，对结果一致）。
- 前端 `delSelected` 改为：单次 POST，response 含 `{ deleted: 142, failed: [] }`。
- 加 `el-progress` 显示请求 → 响应阶段（spinner + 邮件数计数）；服务端响应快
  时进度只是装饰；慢时（>2s）至少有反馈。
- 调用方仍用 `confirmDanger`（v2.29 已加）。

**测试**：新增 `server/__tests__/accounts-batch-delete.test.js` —— 测 50 邮箱
批删一次完成、不存在的邮箱不报错（幂等）、`emails` 字段类型校验。

## 八、风险与回滚

- **HX-9 SIGINT**：在 `process.exit(0)` 之前 await `save.flush()`，如果队列
  卡死会阻塞退出。缓解：加 5s timeout，超时后强制 exit。
- **HX-13 taskkill**：Windows 上某些权限场景 taskkill 会失败。当前 spec 用
  `spawnSync` ignore stdio，失败不抛错；缓解策略是失败时回退到原 `proc.kill()`。
- **PX-7 主动健康检查**：临时切 selector 跑 probe 期间，其他业务请求可能命中
  错节点。缓解：probe 在 `_rotateLock` 内做；probe 完成后必须把 selector 切回
  原 currentNode。
- **FX-13 通知中心**：axios interceptor 误吃业务正常 4xx（如 "/api/execute"
  返 409 表示已运行中），不应推为错误。缓解：interceptor 只对 5xx 自动推；
  4xx 留给业务代码自行调 ElMessage / pushNotification。
- **FX-5 batch-delete**：bulk 事务失败时部分成功部分失败，必须返回详细列表，
  否则用户不知哪些删了。后端实现里逐条 try/catch，response 含 `deleted` 与
  `failed: [{ email, reason }]`。

回滚：与 v2.29 同策略，每项一个 commit；红测试立即 `git revert HEAD`。
