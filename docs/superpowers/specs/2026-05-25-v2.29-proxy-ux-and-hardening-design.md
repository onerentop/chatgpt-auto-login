# v2.29 — 代理 / 前端 UX / 横切硬化优化（设计 spec）

> 日期：2026-05-25  
> 范围：在 v2.28 基础上，**不动登录/支付/JSON 凭证主管道业务逻辑**，对代理子系统、前端人性化操作、以及横切关注点（持久化、日志、启停、安全）做一轮收敛优化。  
> 触发：用户要求"列出当前所有功能 + 找可优化点（含前端后端），登录→支付→JSON 流程不改"。

## 一、背景与目标

v2.28 已经把执行流的可观测性、状态码体系、测活流水线稳定下来。但 v2.18~v2.28 这段时间快速加了 27 份 plan/spec，主代理 / JP 通道 / 黑名单 / 测活 / UI 分组 / 缓存这些子系统功能完整但**横切关注点开始堆积技术债**：

- 代理：`getState()` 全量回吐订阅 URL 含 token；并发 `rotate()` 无锁；`currentNode` 与 sing-box selector 在 `rotationIndex≠0` 时可能不同步；hydrate 与 manual 黑名单的合并不正交。
- 前端：5 个视图功能已收齐，但**人性化短板**集中——筛选不写 URL（不可分享/刷新即失）；删除/中止/清空黑名单的二次确认不统一；Dashboard / Results 漏用 `statusLabel`；侧栏缺 Results 入口；WebSocket 断线不显眼；多处批量操作串行无进度。
- 横切：DB 每次写都 `db.export()` + 非原子 `writeFileSync`；`engine.stop()` 同步函数调 async close 是 fire-and-forget；OTP / access_token 明文进 `server.log` 和 Socket.IO；服务默认监听 `0.0.0.0` 无认证。

目标：在一个版本窗口里清掉这些被忽略的横切短板，并对前端做"看得见的"日常使用人性化升级。**不改业务管道、不改状态机语义**。

## 二、不在范围内（明确排除）

- **登录子系统**：`login.js`、`protocol_register.py`、`chatgpt_register/`、Outlook IMAP / Gmail TOTP 抽取逻辑。
- **支付子系统**：`payment.js` 主流程、`server/chatgpt-checkout.js`、`server/stripe-verify.js`、`stripe_init.py`、`payment-readiness.js`、`server/discord-gateway.js` 业务路径。
- **凭证生成**：`utils.js` 的 PKCE / `saveCPAAuthFile`、`cpa.js`、Codex / Sub2API 格式。
- **状态机与测活语义**：`web/src/status.js` 单一映射；`server/liveness/` 的判定逻辑；`server/db.js` 的 12 个 status / 8 个 alive_status 取值含义。
- **大型架构重构**：迁移到 `better-sqlite3`、`ProxyManager` 类化、移除 `playwright-extra` 等依赖清理。这些价值有限或风险高，留作 v3.0 候选。

涉及上述子系统的发现项**仅修横切短板**（如 OTP / token 在日志里脱敏；引擎 `stop()` 路径资源释放），**不动业务分支判定**。

## 三、当前功能盘点（基线）

> 仅列当前 v2.28 真实启用的功能；已 revert 的 v3 协议扩展、v3 payment 不计。详见 `docs/CHANGELOG.md` 各版本节。

### 登录 / 注册
- Outlook IMAP OTP（imapflow）；Gmail TOTP（otplib）。
- 协议模式 ChatGPT 注册（curl_cffi + chrome131 指纹），完整 Step 0~3。
- 浏览器模式注册（Playwright + CDP 连真 Chrome）。
- access_token + session_json 缓存到 `account_status` 表，跳过 Phase 1 二次登录。
- 协议模式网络错重试 3 次（2s 间隔）+ 代理 fallback。

### 支付管道
- 双支付源：Discord 渠道（hub:chatgpt） / API 渠道（JP-KDDI 直接调 OpenAI checkout）；`paymentLinkSource` 切换。
- Stripe Phase 2.5 零元验证（API 路径才跑）；`no_promo` / `verify_error` 早返回。
- PayPal 12 字段自动填表 + SMS 验证（自动调外部 `smsApiUrl`）。
- 支付链接缓存到 `account_status.payment_link`，重试跳过 checkout。
- Stop 按钮 → `io.emit abort` → `abortableSleep` 秒级响应。

### 代理子系统
- V2Ray/Reality 订阅解析（vmess/vless/trojan/ss/hysteria2）。
- sing-box 双 mixed inbound：`7890` 主代理 / `7891` JP 通道。
- Clash API 切 selector：`auto-rotate` / `jp-checkout`。
- 主代理按 `regionFilter`（默认 US）或 `whitelist` 精确选；JP 按 `keyword`（默认 KDDI）或 `jpCheckout.whitelist` 精确选。
- 节点黑名单：3 连败拉黑 30 min、跨重启持久化、`source` 区分 `auto`/`manual`。
- 轮换游标：`refresh()` 保留 `rotationIndex`，拉黑后 fire-and-forget rotate。
- 出口探测：`detectExit` / `detectJpExit`。
- 节点信息透传：`proxy_node` / `exit_ip` 写 `account_status`，UI reason 列展示。
- `bin/sing-box.exe` 自下载 + 版本探活。

### 凭证生成
- PKCE OAuth（协议 HTTP / 浏览器 Chrome），`refresh_token` 失败标 `plus_no_rt`。
- `cpa-auth/codex-{email}.json`（Codex 格式）+ `sub2api-{email}.json`（Sub2API 格式）。
- 可选 CPA 外部注册（`cpa.js`，多数场景不用）。

### 持久化 / 状态机
- 三张表：`accounts` / `account_status` / `execution_logs`；sql.js WASM。
- 12 个 status / 8 个 alive_status；`web/src/status.js` 单一映射。
- `statusDB.set()` merge-aware（不传字段保留旧值）。
- 测活终态映射回 status；plus_no_rt + alive=plus 时保留 plus_no_rt。

### Web 仪表盘
- Dashboard：5 KPI 卡 + 流水线进度条 + 最近执行表。
- Accounts：增删改、批量导入、5 维度搜索、多筛选、列内复制、测活全部 / 选中。
- Execute：按状态分组折叠、实时日志展开行、reason 列、proxy_node / exit_ip banner、Stop。
- Results：历史结果表、status 筛选、批量下载凭证（CPA / Sub2API）。
- Config：7 tabs（支付 / 执行 / Discord / OAuth / 代理 / JP / 节点黑名单）。
- Socket.IO 实时推 `log` / `account-status` / `complete`。

## 五、v2.29 本轮落地范围

按"高 impact + 可控 risk + 一次会话内可完成 + 不动业务管道"四条筛选，本轮做以下 **20 项**：

### 5.1 代理（4 项）

- **PX-1（P0）订阅 token 脱敏**  
  `server/proxy/index.js#getState()` 改为显式字段白名单 return；新增 `hasSubscription: !!_state.subscriptionUrl`、`subscriptionHost: <hostname>`。前端 Config.vue 改读这两个字段。
- **PX-2（P0）rotate 串行化**  
  在 `server/proxy/index.js` 加 module 级 `_rotateLock = Promise.resolve()`，把 `rotate / rotateJp / switchTo / refresh` 的关键临界区（涉及 `clashApi.switchSelector` 与 `_state.currentNode` 写入）串到锁里。`recordBadAttempt` 拉黑后不再 fire-and-forget rotate；改为返回 `{ blacklisted: true, requestRotate: true }`，由调用方决定何时显式 await rotate（同时把现在 `engine.js:493` / `protocol-engine.js:283` 的 catch 调用维持，因为它本来就在显式 await，不需要再加 fire-and-forget）。
- **PX-3（P0）`currentNode` 与 selector 同步**  
  `refresh()` 末尾在 `rotationIndex !== 0` 时主动 `await clashApi.switchSelector(SELECTOR_TAG, _state.currentNode)`（JP 同理），并在锁内执行。
- **PX-5（P1）all-bad clear 区分 auto / manual**  
  `rotate()` 兜底分支只清 `source === 'auto'`；如仍无可用节点再退化为全清。`web/src/views/Config.vue` 的黑名单 tab 给 `auto` / `manual` 来源加视觉区分（已有 `source` 字段）。

不做：PX-4 stop-after-start-success（要重写 singbox.js lifecycle，风险高）、PX-7 主动健康检查（要新 Clash API 探测代码）、PX-8 secret + 端口自动分配（要联动 sing-box config 生成 + 测试夹具）、PX-9~15（测试盲区、性能、SSRF、代码重复均为 follow-up）。

### 5.2 前端（10 项）

- **FX-1（P0）Dashboard / Results 用 `statusLabel`**  
  `Dashboard.vue:58`、`Results.vue:35` 改为 `{{ statusLabel(row.status) }}`，`el-tag` 的 `type` 用 `statusTagType(row.status)`（在 `status.js` 暴露的 `TYPE_MAP` 之上加一个辅助函数）。
- **FX-2（P0）侧栏菜单顺序 + 加 Results 入口**  
  `AppLayout.vue` 顺序改为：仪表盘 / 账号管理 / 执行控制 / 执行结果 / 配置设置。
- **FX-3（P0）WebSocket 断线显眼提示**  
  `App.vue`（或 `AppLayout.vue`）加 sticky 顶部 `el-alert` 横幅，监听 `socket.disconnected`，提示 "实时数据已暂停，正在重连…" + 手动 "重连"按钮。
- **FX-6（P0）Config dirty form 守卫**  
  `watch(form, () => isDirty = true, { deep: true })`，`onBeforeRouteLeave` + `window.addEventListener('beforeunload')` 在 `isDirty` 时弹确认。
- **FX-7（P0）键盘快捷键基础三键**  
  `/` 聚焦当前页搜索框、`Esc` 关弹窗（Element Plus 默认有，补 dialog/drawer 缺的）、`Ctrl+Enter` 提交主表单（Accounts 添加、Config 保存）。封装到一个 `useHotkeys.js` composable。
- **FX-8（P0）筛选写 URL（Accounts 起手）**  
  Accounts.vue 把 `search / status / plan / auth / alive / stale` 同步到 `router.replace({ query })`，进入页面时从 query 反读；以后 Execute / Results 沿用同一 helper（本轮只先做 Accounts，留接口给后续）。
- **FX-9（P1）全 `FAILED_TO_RETRY_STATUSES` 行内重试**  
  `AccountTableRows.vue` 单行重试按钮的显示条件从 `error / idle` 扩到 `isFailedToRetry(row.status)`。
- **FX-11（P1）二次确认统一**  
  把 `Accounts.vue` 删除选中、Execute 停止、Config 清空黑名单全部统一到 `confirmDanger(message, { requireText?, type })` helper（基于 `ElMessageBox.confirm` / `.prompt`），高危（删账号 >5、清空黑名单）要求输入数字 N 才确认。
- **FX-12（P1）Results 状态下拉用 `EXECUTE_STATUS_FILTER_OPTIONS`**  
  从 `status.js` 导入同一份；用一个 `RESULTS_STATUS_FILTER_OPTIONS` 别名表达"含活性中间态"的差异点。
- **FX-14（P1）Config 黑名单 polling 在 tab 不可见时暂停**  
  `document.visibilityState === 'hidden'` 时 `clearInterval`，可见时重启。

不做：FX-4 Pipeline HUD（大；下个版本单独 spec）、FX-5 批删进度 / 后端 batch-delete（要后端新 endpoint）、FX-10 Execute toolbar 重排（视觉重做）、FX-13 通知中心、FX-15 长字段折叠列（视觉重排）、FX-16~19（次要）。

### 5.3 横切（6 项）

- **HX-1a（P0）DB 原子写**  
  `server/db.js#save()` 改为 `writeFile(tmp) → renameSync(tmp, DB_PATH)`；`fs.renameSync` Windows 同盘原子。同时把同步 `writeFileSync` 改为 `fs.promises.writeFile` + 串行化（结合 HX-8）。
- **HX-1b（P0）DB 写串行化**  
  模块级 `let _saveQueue = Promise.resolve()`；`save()` 改为 `_saveQueue = _saveQueue.then(actualSave)` 返回它的 promise。所有调用方 `await db.save()` 现在的同步用法变成 await；但 sql.js 的 mutation 仍同步，只是落盘异步。
- **HX-2（P0）日志脱敏 — LogCapture 兜底**  
  在 `server/logger.js#LogCapture` 处理消息前过一层 `redact()`：JWT (`eyJ...` 三段)、6/8 位 OTP 紧邻 `otp|code|verification` 关键词、`access_token=` / `refresh_token=` URL 参数、PayPal SMS 验证码。`utils.js` / `protocol-engine.js` / `login.js` 等位置的 console.log 维持现状（不动业务管道），由 LogCapture 兜底替换。
- **HX-3（P0）`engine.stop()` async + start 路由 await**  
  `server/engine.js#stop()` 和 `protocol-engine.js#stop()` 改 `async`，串行 `await` close 各资源；`server/routes/execute.js` 的 start 处理 `if (engine) await engine.stop();`。
- **HX-6（P0）`config.json` 原子写**  
  `server/routes/config.js` 写盘改 `tmp + rename` 模式（同 HX-1a）。
- **HX-7（P0）默认绑 `127.0.0.1` + `HOST` 环境变量**

## 九、风险与回滚

- **PX-1 字段移除**：如果前端有未被识别的代码消费 `subscriptionUrl`，会 silently undefined。缓解：grep `subscriptionUrl` 全前端代码，确认无消费方；保留 `hasSubscription` 兼容。
- **HX-1b 写串行化**：现存调用方都把 `db.save()` 当同步看待，改 async 后忘 await 会丢写。缓解：grep `\.save\(\)` 全局过一遍，逐个加 `await`；测试覆盖关键路径。
- **HX-7 绑 127.0.0.1**：如果用户原本在跨机器访问（即使没做安全考虑），会突然访问不到。缓解：`start.bat` 启动横幅显示监听地址；README 说明 `HOST` 环境变量。
- **PX-2 rotate 锁**：如果锁住的临界区里有 await 长任务，会阻塞其它账号。缓解：锁内只放 selector 切换 + `_state.currentNode` 写入，订阅 fetch 等长耗时操作不进锁。

回滚：每个 P0 改动单独提交，CI 全绿后再下一项；失败立即 `git revert <commit>`。
