# v2.36 — GitHub 工具风换皮（设计 spec）

> 日期：2026-05-25  
> 范围：用户反馈 v2.35 indigo + Linear/Vercel SaaS 风格不喜欢。本轮**完全
> 换一种主题**：GitHub 风（方正、信息密度高、灰蓝绿调、几乎无阴影、更
> 偏开发者工具）。只换 CSS 层 + 几处硬编码 brand 引用；**所有功能 /
> 布局结构 / 业务逻辑保持 v2.35 不动**。  
> 触发：用户原话"现在的前端风格我不喜欢，换一种"。

## 一、为什么换不动结构

v2.34（design tokens + 包壳）和 v2.35（任务流 + Command Palette +
Pipeline HUD + ContextActionBar）已经把 IA 和组件做到位了。问题在
"风格"，不在"结构"。换风格只需：

- 替换"主题层"CSS（`v235-polish.css` → `v236-github.css`）。
- 修若干视图里直接引用 `--el-color-primary-light-9` 等 indigo 派生
  色的地方，让它们读新主题 token。

不动：路由 / 组件 / 视图业务 / composables / store / 后端。

## 二、目标风格对比

| 维度 | v2.35 SaaS 风（现状） | v2.36 GitHub 工具风（目标） |
|------|---------------------|---------------------------|
| 主色 | indigo `#5b67f0` | GitHub blue `#0969da` |
| 成功色 | Element default 绿 | GitHub green `#1f883d` |
| 警告 / 危险 | Element 默认 | GitHub `#9a6700` / `#cf222e` |
| 主背景 | 冷白 `#f8f9fc` | GitHub `#ffffff`（主区）+ `#f6f8fa`（次区 / 侧栏） |
| 边框 | `#e2e6ee` 偏蓝灰 | `#d0d7de` GitHub 中性灰（更明显） |
| 圆角 | sm/md/lg 6/10/14 | 6/6/8（更克制，"github 卡片"感） |
| 阴影 | 多层 sh-1/2/3（几乎察觉不到） | **几乎无阴影**，全靠 1px border 区分层级 |
| 字体 | system stack | system stack + `Mona Sans, Inter, ...` |
| 字号 | 13~28 | 12~24（更紧凑） |
| 按钮 hover | translateY -0.5px | 仅 background 变化，无位移 |
| 焦点环 | 2px brand outline | 2px brand outline + 1px transparent inner |
| 状态色（行高亮）| 浅黄/浅绿/浅红/浅灰 | 同色家族但偏 cooler（GitHub 风的浅色） |

GitHub 风核心特点：**克制 + 信息密度 + 看着像"工程工具"而不是"创业
产品"**。

## 三、本轮范围

### 3.1 做（3 件事）

1. **B1 — 写 `web/src/styles/v236-github.css`**：新主题层覆盖 v2.35
   polish 的主要变量。Element Plus 全局样式也覆盖：去 hover 位移、
   去多层阴影、用 1px border 替代、tag/card/dialog 圆角调小。
2. **B2 — 改若干硬编码 indigo 引用**：grep `--el-color-primary-light-9`
   / `91, 103, 240` 之类的 SaaS-flavored 色片，换成新主题 token 或
   保持 brand-light-9 但 GitHub 风下它会自动变浅蓝色。
3. **B3 — `main.js` 引入新主题** + CHANGELOG 记录 + 重启服务。

### 3.2 不做

- 后端 / API / status.js / 组件结构 / Command Palette / Pipeline HUD
  等全 v2.35 ship 项的行为。
- 引入新 npm 依赖。
- 推翻 v2.34 tokens.css 或 v2.35 polish.css 文件本身 —— 保留它们
  让用户能 `git revert` 回到任意风格。
- 切换字体到非 system（无网络请求；用 macOS / Windows 内置）。

## 五、需要适配的硬编码 indigo 引用

通过 grep 找到以下点（v2.35 各组件用了 `--el-color-primary-light-9`
作为"很浅的 brand bg"——GitHub 主题下它自动变浅蓝，无需修改）：

- `web/src/components/AppLayout.vue`：`--app-sidebar-item-active-bg`、
  `app-topbar__cmd` 的 kbd 背景；这些已经引用 token，自动跟随。
- `web/src/components/ui/CommandPalette.vue`：`.cmd-item--active` 用
  `--el-color-primary-light-9`；自动跟随。
- `web/src/components/ui/PipelineHUD.vue`：`.hud` border 用 `--el-color-primary-light-7`、`.hud__spinner` 背景用 `--el-color-primary-light-9`；自动跟随。
- `web/src/components/ui/ContextActionBar.vue`：`.cab__count strong` 用 `--app-brand`；自动跟随。
- `web/src/views/Dashboard.vue`：`.hero__icon--brand` 和 `.task__icon--brand` 用 `--el-color-primary-light-9` 背景；自动跟随。
- `web/src/components/AccountTableRows.vue`：v2.33 `.row-status-running` 用 `rgba(64, 158, 255, 0.10)` 写死蓝色 → **改为 `var(--app-row-running)` token**。

## 十、风险与回滚

- **风险**：完整覆盖 Element Plus brand 后某些组件（如 el-radio
  active、el-checkbox 勾选、el-link 主色）可能视觉不和谐。缓解：
  Element Plus 默认就用 `--el-color-primary-*` 派生，token 替换会
  自动协调。如有局部 broken，下个 patch 单独修。

回滚：`git revert HEAD~2..HEAD` 回到 v2.35 indigo。`v235-polish.css`
未删除，回滚后无依赖问题。

## 附录 A：决策记录

- **为什么选 GitHub 风**：技术运营用户、工具属性、信息密度需求高、
  长期使用不疲劳。蓝绿配色比 indigo 更"中性"。比 brutalist 不显
  hacker、比 Material 不显消费品。
- **为什么保留 v235-polish.css**：用户口味不一定接受 GitHub 风；保
  留前一版让 `git revert` 一次就能回去；以后如果想再换 (如 Notion
  暖白风)，只需复制 v236 文件改改。
- **为什么不动暗色变量太多**：GitHub 暗色 `#0d1117` 是公认护眼的；
  Element Plus 暗色 + 我们的暗色 token 已经够好。
