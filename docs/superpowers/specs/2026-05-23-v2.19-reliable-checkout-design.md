# v2.19 Reliable JP-First Checkout Design

**Date:** 2026-05-23
**Status:** Draft → 待评审
**Predecessors:** v2.18.0 (checkout-via-jp-kddi), v2.18.1 (jp-checkout-whitelist), v2.18.2 (country=US/USD 1-line)

## Background

v2.18.2 实验阶段发现 3 个关键事实，与之前假设相反：

1. **JP IP 是 OpenAI 给 plus-1-month-free coupon 的必要条件**。同一账号、同一 body（country=US, currency=USD, promo=plus-1-month-free），从直连/US 出口调用拿 $20，从 JP-KDDI 出口调用拿 $0。
2. **promo 资格不是一次性消耗**。之前误认为"调一次 checkout 就吃掉资格"，实际是直连没走 JP IP 时 OpenAI 不给 coupon。用 JP IP 重新调，有资格账号仍能拿 $0。
3. **bot 的 Eligibility Check 真实有效**。被 bot 标记"无资格"的账号（gexi4056685、qnke4812473），即使强制 JP IP 调用也只拿 $20。OpenAI 服务端确实维护 per-account 资格 flag。

**当前实现的隐蔽问题**：

- `server/chatgpt-checkout.js:14` 的 `const proxy = jpUrl || mainUrl` 静默回退。当 JP 节点池空/未启用时回退到主代理（US 出口），结果拿到的是 $20 link，但流程继续进入 payment.js 浪费资源，最终在 PayPal 阶段失败标记为 `status=error/phase=payment`。
- engine 没有验证 link 的 amount_due，所有 $0/$20 link 同样处理。
- 用户和运营无法从 dashboard 区分"账号无资格"、"JP 节点临时不可用"、"Stripe 暂时不可达"这三种不同语义的失败。

## Goal

让 v2.19 的 checkout 阶段实现**确定性 fail-fast**：要么拿到经 Stripe 验证的 `$0` link，要么尽早停止该账号，永远不浪费 payment.js。

## Architecture

### Data Flow (with fail-fast exits)

```
Phase 2: checkout API
  ├─ proxy = proxyMgr.getJpProxyUrl()
  ├─ if (!proxy)  →  account.status = no_jp_proxy ⛔ STOP
  ├─ call OpenAI /backend-api/payments/checkout via 7891 JP
  ├─ if (no link)  →  account.status = no_link (现有行为) ⛔ STOP
  └─ ↓ got link

Phase 2.5: Stripe init 验证（v2.19 新增）
  ├─ extract cs_id from link
  ├─ call https://api.stripe.com/v1/payment_pages/{cs_id}/init via main proxy (7890 US)
  ├─ parse invoice.amount_due
  ├─ if (amount != 0)        →  account.status = no_promo ⛔ STOP (保留 link)
  ├─ if (init fetch fail)    →  account.status = verify_error ⛔ STOP (保留 link)
  └─ ↓ verified $0

Phase 3: payment.js（不变）
  └─ Chrome 打开 link，自动 PayPal 流程
```

### 三条新 fail-fast 路径

| 状态 | 触发条件 | 含义 | 可重试 |
|---|---|---|---|
| `no_jp_proxy` | `proxyMgr.getJpProxyUrl()` 返回空 | JP 节点池暂时不可用 | ✅ JP 恢复后 |
| `no_promo` | `invoice.amount_due > 0` | OpenAI 服务端判定该账号无资格 | ❌ 账号资格问题 |
| `verify_error` | Stripe init 调用失败（timeout / 4xx / 解析失败）| Stripe 暂时不可达或拒绝 | ✅ Stripe 恢复后 |

## Component 1: `server/chatgpt-checkout.js` — 强制 JP

### 现状（v2.18.2）

```js
const jpUrl = proxyMgr.getJpProxyUrl();
const mainUrl = proxyMgr.getProxyUrl();
const proxy = jpUrl || mainUrl;       // ⚠️ 静默回退
const usingJp = !!jpUrl;
// ... 出错时 raw 里加 "WARN: jp_channel_disabled, fallback..."
```

### v2.19 改动

```js
const jpUrl = proxyMgr.getJpProxyUrl();
if (!jpUrl) {
  resolve({
    link: '',
    title: '',
    raw: 'NO_JP_PROXY: JP checkout channel unavailable',
    noJpProxy: true,                  // 新字段，engine 据此设状态
  });
  return;
}
const proxy = jpUrl;                  // 不再回退
```

### 清理掉的死代码

- `mainUrl` 变量（不再读）
- `usingJp` 变量（恒为 true）
- raw 里 "WARN: jp_channel_disabled" 警告分支（不再触发）

### 单元行为

- **输入**: `accessToken`, `opts.country/currency/promoCampaignId`
- **输出**（3 种）:
  - `{ link: <url>, raw: ... }` — 拿到 link，**未验证 amount**
  - `{ link: '', noJpProxy: true, raw: 'NO_JP_PROXY...' }` — JP 不可用
  - `{ link: '', raw: 'ERROR: ...' }` — Python/curl_cffi 失败

### 不变的

- 60s timeout
- markJpBad 节点淘汰
- impersonate 多浏览器随机轮换

## Component 2: `server/stripe-verify.js`（新增）

### Interface

```js
async function verifyCheckoutIsFree(payLink) → {
  ok: boolean,
  is_free: boolean,        // amount_due === 0
  amount_due?: number,
  currency?: string,
  has_paypal?: boolean,
  coupons?: string[],
  reason?: string,         // 'invalid_link' | 'stripe_init_timeout' | 'stripe_init_403' | 'no_invoice'
}
```

### 实现路径

#### Step 1: 从 link 提取 cs_id

```js
const m = payLink.match(/\/c\/pay\/(cs_live_[A-Za-z0-9]+)/);
if (!m) return { ok: false, reason: 'invalid_link' };
const csId = m[1];
```

#### Step 2: 调 Stripe init API via curl_cffi（走 main proxy 7890 US）

```py
# stripe_init.py — 仿照 checkout_link.py 的 spawn 协议
url = f"https://api.stripe.com/v1/payment_pages/{cs_id}/init"
r = cr.post(url, data={"eager_browser_locale": "en-US", "expected_amount": ""},
            impersonate=imp, proxies=proxies, timeout=15)
```

**Stripe 拒绝时的 fallback 策略**（实施时按需启用）:
1. 先 GET pay.openai.com checkout page，提取 `pk_live_xxx` publishable key
2. 用 `key=pk_live_xxx` 作为 form param 再调 init
3. 终极兜底：Chrome 渲染（沿用 payment.js 现有逻辑）

#### Step 3: 解析 invoice

```js
const inv = data.invoice || {};
return {
  ok: true,
  amount_due: inv.amount_due,
  currency: inv.currency,
  is_free: inv.amount_due === 0,
  has_paypal: (data.payment_method_types || []).includes('paypal'),
  coupons: (inv.total_discount_amounts || []).map(d => d.coupon?.name).filter(Boolean),
};
```

### 出口代理选择

**main proxy (7890 US)**，因为：

- cs_live 创建后任何 IP 都行（实验已证）
- 走 US 出口让 Stripe 默认填美国 billing，跟最终支付 IP 一致
- JP 节点是稀缺资源，留给 checkout API 用

### 文件结构

- `server/stripe-verify.js` — JS 包装层，与 `chatgpt-checkout.js` 平行
- `stripe_init.py` — Python 实现层（curl_cffi）

## Component 3: `server/engine.js` Phase 2.5 集成

### 插入位置

Phase 2 完成（fetchCheckoutLink 的 retry 循环结束）与 Phase 3（payment.js / autoPayment 调用）开始之间。当前 engine.js 大约 line 313 — 实施时以 `if (discord.link)` 那一行的上下文为锚点。

### 新的分支结构

```js
// === Phase 2 完成后 ===
if (discord.noJpProxy) {
  // 分支 A: JP 不可用 → no_jp_proxy
  console.log(`${p} No JP proxy — skipping`);
  finalResult = {
    email: account.email,
    status: 'no_jp_proxy',
    paymentLink: '',
    reason: 'JP checkout channel unavailable'
  };
}
else if (!discord.link) {
  // 分支 B: 拿不到 link → no_link（现有行为）
  finalResult = { email: account.email, status: 'no_link', paymentLink: '', reason: discord.raw.slice(0, 200) };
}
else {
  // 分支 C: 拿到 link → Phase 2.5 验证
  currentPhase = 'verify';
  console.log(`${p} Phase 2.5: Verifying $0 via Stripe init...`);
  this.emitStatus({ email: account.email, status: 'running', phase: 'verify', progress });
  const v = await verifyCheckoutIsFree(discord.link);

  if (!v.ok) {
    // C1: Stripe init 调用失败
    finalResult = {
      email: account.email,
      status: 'verify_error',
      paymentLink: discord.link,
      reason: `Stripe init: ${v.reason}`
    };
  }
  else if (!v.is_free) {
    // C2: link 不是 $0
    console.log(`${p} Not free: amount_due=${v.amount_due} ${v.currency}`);
    finalResult = {
      email: account.email,
      status: 'no_promo',
      paymentLink: discord.link,
      reason: `amount_due=${v.amount_due} ${v.currency}`
    };
  }
  else {
    // C3: 验证通过 → 继续 Phase 3（现有代码）
    console.log(`${p} ✓ Verified $0 + coupons=[${v.coupons.join(',')}]`);
    finalResult = { email: account.email, status: 'error', paymentLink: discord.link, reason: '' };
    currentPhase = 'payment';
    // ... 原有 Phase 3 + Phase 4 代码块（line 320-394）整体不变，被这个 else 包住
  }
}
```

### 关键不变点

- Phase 2 的 3× retry 逻辑保留
- Phase 3 (payment.js + autoPayment) 代码块整体不动，只是被 if 包住
- Phase 4 (OAuth + CPA) 代码不动

### Summary 统计

新增 3 个 bucket：

- `summary.noJpProxy` (匹配 `no_jp_proxy`)
- `summary.noPromo` (匹配 `no_promo`)
- `summary.verifyError` (匹配 `verify_error`)

## Testing

### 单元测试

#### `fetchCheckoutLink` 强制 JP

- **case A**: `proxyMgr.getJpProxyUrl()` 返回非空 → 正常调用 Python，传 JP proxy
- **case B**: `getJpProxyUrl()` 返回 `''` → 立即 resolve `{ link: '', noJpProxy: true }`，**不启动 Python 进程**
- **case C**: Python timeout → resolve `{ link: '', raw: 'ERROR: Python timeout' }`

#### `verifyCheckoutIsFree`

- **case A**: 有效 cs_live URL + Stripe 返回 amount_due=0 → `{ ok: true, is_free: true, has_paypal: true, coupons: [...] }`
- **case B**: amount_due=2000 → `{ ok: true, is_free: false, amount_due: 2000 }`
- **case C**: 非法 link (无 cs_live)  → `{ ok: false, reason: 'invalid_link' }`
- **case D**: Stripe init 超时 → `{ ok: false, reason: 'stripe_init_timeout' }`
- **case E**: Stripe 返回 403/401 → `{ ok: false, reason: 'stripe_init_403' }`

## Acceptance Criteria

1. ✅ `fetchCheckoutLink` 在 JP 不可用时立即返回 `noJpProxy:true`，**不启动** Python 子进程
2. ✅ engine 在 Phase 2.5 拦截所有 `amount_due > 0` 的 link，账号被标 `no_promo` 而非 `error`
3. ✅ Stripe init 网络失败时账号被标 `verify_error` 而非 `error`
4. ✅ 验证通过的 $0 link 才进入 payment.js
5. ✅ Web Dashboard 显示新 3 状态，徽章颜色和文案正确
6. ✅ 端到端跑 4 个测试账号，状态分布符合预期
